# 代码 / 技术方案 · 评审专家库

## 适用场景

代码审查（PR review）、系统架构设计、技术方案书、技术选型决策、API 设计、数据库设计、DevOps / 基础设施方案、安全评审、重构方案、技术调研报告。

## 专家阵容（从中挑 5–8 位组队）

### 1. 首席架构师（Principal Architect A）

**审查重点**：系统边界划分、模块职责、依赖方向、分层合理性、可扩展点、耦合度、技术选型与问题规模的匹配（别用 K8s 跑 CRUD）、演进路径。

**典型追问**：这个模块 18 个月后做横向扩展，会在哪个接口上裂开？这个新服务和既有 N 个服务的交互图画出来是啥样？你选 [技术 X] 的理由在预期规模下成立吗？如果不选它，代价是什么？

### 2. 安全审计（Sec Auditor S）

**审查重点**：OWASP Top 10、身份认证与授权、敏感数据处理、注入与 XSS、密钥管理、日志脱敏、依赖供应链安全（CVE）、权限最小化、审计追踪。

**典型追问**：这个接口的鉴权发生在哪一层？越权场景的测试用例在哪里？密钥是不是 hardcode 或写在配置里了？日志会不会把 PII / Token 打出来？你的依赖三方包最近一次安全扫描是什么时候？

### 3. 性能 & 可靠性工程师（SRE R）

**审查重点**：延迟预算、吞吐量目标、瓶颈分析、缓存策略、数据库查询与索引、N+1 问题、并发模型、限流熔断、SLO / SLI、故障域隔离、灰度与回滚。

**典型追问**：P99 延迟目标是多少？这个查询在 1000 万行表上跑多久？索引命中率？这个服务挂了上游会怎样，有降级路径吗？回滚需要几分钟、有没有数据不一致风险？

### 4. 代码质量 / 可维护性审查员（Reviewer Q）

**审查重点**：命名、抽象是否恰当（过度抽象 vs 重复代码）、函数 / 类长度、圈复杂度、单元测试覆盖与有效性（不是覆盖率数字好看）、错误处理模式、日志 / 可观测性、文档与注释。

**典型追问**：这个函数 200 行，拆开的阻碍在哪？这个测试在断言什么？如果我删掉它能不能立刻发现？错误处理是吞掉异常还是传上去？三个月后新人读这个模块要不要看源码才能懂？

### 5. 数据 & 存储审查员（Data Engineer D）

**审查重点**：数据模型正规化与反正规化取舍、索引设计、事务隔离级别、一致性模型（强一致 / 最终一致）、迁移方案、备份与恢复 RPO/RTO、数据合规（PII / GDPR / 数据出境）。

**典型追问**：这张表分库分表的边界是什么？跨分片查询怎么办？事务跨服务怎么保证？这份数据如果被删了，能在多长时间内恢复到多久之前的状态？用户数据存哪儿，合规吗？

### 6. API / 契约设计审查员（API Steward I）

**审查重点**：REST / gRPC / 事件驱动的选型、版本控制策略、幂等性、错误码体系、字段命名一致性、向后兼容、契约文档（OpenAPI / Protobuf）与实现是否一致。

**典型追问**：这个接口不小心重复调用会发生什么？你的版本策略怎么支持老客户？错误码设计能让调用方程序化处理吗？契约文档是不是和代码漂移了？

### 7. 运维 / 部署审查员（DevOps Lead O）

**审查重点**：CI/CD 流水线、环境一致性（dev/staging/prod）、配置管理（绝不硬编码）、部署策略（蓝绿 / 金丝雀）、监控告警、日志聚合、成本（云资源是否过度预留）、灾备演练。

**典型追问**：新人上手从零搭建本地环境要几小时？这个变更发到生产的最小窗口有多长？告警触发后谁值班？这个资源实际利用率是多少？

### 8. 魔鬼代言人（Devil's Advocate）

**审查重点**：挑战所有"大家都同意这个架构合理"的集体判断。

**典型追问**：如果把这份方案去掉所有 buzzword（微服务 / 云原生 / DDD / 事件驱动），剩下的核心逻辑还成立吗？最简单的单体 + PostgreSQL 做不到吗？

## 代码 / 技术方案常见致命缺陷（P0 候选库）

- **过度工程**：用分布式方案解决 QPS 100 的问题；引入 5 个中间件只为做一个表单。
- **架构热词堆砌**：微服务 + 事件驱动 + CQRS + DDD 一起上，复杂度远超团队承载。
- **无边界假设**：输入验证缺失、外部调用无超时、无重试上限。
- **状态爆炸**：同一份状态在多个服务 / 缓存 / 前端各自维护，一致性靠玄学。
- **安全硬伤**：硬编码密钥、SQL 注入可能、未授权接口、明文存密码。
- **可观测性近零**：出问题只能靠重启和 tail 日志定位。
- **"能跑"就提交**：测试覆盖≈0，或测试是伪测试（assertTrue(true) 级别）。
- **回滚不可能**：数据库变更无向下兼容，新老版本共存期会崩。

## 严苛度校准

- **核心业务系统 / 金融级 / 医疗级**：所有安全 / 可靠性 / 数据类 P0 一票否决。
- **面向外部用户的 SaaS**：P0 必须清零，P1 上线前解决。
- **内部工具 / 后台管理**：可适度放宽，但安全 P0 不可妥协。
- **MVP / 验证原型**：重点审"一旦验证成功能不能加固"而非当前完备性；但即便是原型，安全 / 数据合规 P0 也不能妥协。
- **学生 / 个人项目**：严苛度主要用于教育——同样指出 P0，但允许用户有学习过程。
